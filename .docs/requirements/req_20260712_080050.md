# System Specification: Clean Architecture & Best Practices Refactoring

## 1. Escopo e Intenção
A intenção principal é refatorar o projeto atual (Skill Optimizer MCTS) para que ele se torne um projeto "limpo, seguindo as melhores práticas sem perder a qualidade". Isso implica em aplicar os princípios do **Design by Contract**, **SOLID**, e **Clean Architecture**, reduzindo o acoplamento entre os módulos, clarificando as responsabilidades de cada entidade e definindo interfaces estritas.

## 2. Atores e Entidades do Domínio

### Atores
- **Pipeline de Execução (Runner):** Coordena o ciclo de vida da otimização.
- **Avaliador Externo (LLM/Juiz):** Provê feedback sobre o desempenho ou a semântica gerada.

### Entidades Core
- **MCTSNode / MCTS Tree:** Representa o estado da busca, mantendo estatísticas (visitas, recompensas).
- **Optimizer:** O motor principal que orquestra a busca, expansão e simulação.
- **Evaluator (Heuristic, Semantic, Density):** Responsável por atribuir pontuações a um nó específico.
- **Mutator / MutationStrategy:** Aplica transformações aos nós ou prompts para gerar novos estados.
- **ExperienceStore:** Gerencia o armazenamento persistente de trajetórias, resultados e telemetria.
- **DriftMonitor:** Monitora e avalia o desvio de performance/comportamento ao longo do tempo.

## 3. Regras de Negócio Inflexíveis (Condições Binárias)
1. **O motor de otimização (Optimizer) DEVE depender apenas de interfaces (abstrações) de Evaluators e Mutators, NUNCA de implementações concretas.** (Sim/Não)
2. **Nenhuma camada de domínio (Core Entities) DEVE depender de frameworks externos de roteamento HTTP (ex: FastAPI) ou bibliotecas de I/O diretas.** (Sim/Não)
3. **Todo dado que entra ou sai de um contrato DEVE ser validado através de schemas estritos.** (Sim/Não)
4. **O estado do nó (MCTSNode) DEVE ser imutável após a sua avaliação consolidada, exceto para atualizações atômicas de heurística de backpropagation.** (Sim/Não)
5. **O ExperienceStore DEVE ser acessado através de uma interface de repositório genérica, permitindo troca do backend de persistência sem impacto no domínio.** (Sim/Não)

## 4. Critérios de Aceite Estritos
- O código-fonte refatorado deve passar no linter e no type checker (`mypy --strict` ou equivalente) sem ignorar tipagens.
- Existência de interfaces definidas via `typing.Protocol` ou `abc.ABC` para todos os componentes intercambiáveis (Avaliadores, Estratégias de Mutação, Repositórios).
- Cobertura de testes unitários para a camada de Domínio operando isolada de serviços externos.
- Separação física e lógica da infraestrutura (APIs, leitura de disco) do núcleo de otimização.

## 5. Elementos Fora de Escopo
- Alterações nos algoritmos fundamentais do MCTS ou fórmulas de avaliação, caso os mesmos já operem conforme esperado.
- Alteração ou desenvolvimento de interface de usuário (Frontend).
- Migração de provedor de LLM ou mudanças drásticas na lógica de prompts, exceto se estritamente para adequação aos contratos definidos.
- Modelos de banco de dados físicos ou ORM complexos (foco apenas no contrato/interfaces).
